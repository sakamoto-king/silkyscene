# Scene v2 思维模型与实现路线

本文档用于锁定 SilkyScene 的下一代场景系统方向，作为实现与评审的共同基线。

目标不是局部打补丁，而是让 @silkysite/scene 从“可演示”走向“可发布、可维护、可扩展”。

---

## 1. 背景与问题

现状存在以下可维护性与性能问题：

1. 场景状态书写重复严重，开发者需要频繁复制整段状态。
2. 布局只支持显式 absolute 百分比声明，表达复杂结构时可读性较弱。
3. 运行期存在不必要的布局决策压力，不利于动画稳定帧率。
4. 父子关系难以在场景间动态表达。
5. 缺乏标准导入导出协议，不利于生态工具链。

---

## 2. v2 核心原则

1. 默认增量状态：场景只写变化字段，未声明字段自动继承。
2. 结构先表达、再编译：允许使用自然语言布局描述，再统一解算为绝对百分比布局。
3. 运行期轻量化：复杂布局计算在启动或重编译阶段完成，切场景仅做状态读取和插值。
4. 场景级父绑定：同一元素可以在不同场景绑定不同父对象。
5. 导入导出优先源码态：先保证可编辑、可回放，再扩展编译态。
6. 本次升级是 Breaking Change：优先模型一致性，不以兼容旧 API 为第一目标。

---

## 3. 状态模型升级

### 3.1 默认增量语义

Scene.setState(element, state) 在 v2 中默认执行深合并：

- 当前场景同元素已有状态：与新状态深合并。
- 当前场景同元素无状态：以新状态建立初值。
- 场景最终态仍通过 Presentation 的 resolved cache 按场景顺序继承得到。

含义：开发者只需描述“相对上一状态的变化量”。

### 3.2 全量替换入口

新增显式全量替换 API（例如 Scene.replaceState），用于以下场景：

1. 需要彻底覆盖上一场景残留字段。
2. 需要明确重置某段复杂子对象。

### 3.3 数组合并策略

数组字段统一采用整段替换，不做按索引深合并。

原因：

1. 语义确定，避免隐式拼接导致的不可预期结果。
2. 更利于导入导出时的跨语言一致性。

---

## 4. 布局模型升级

### 4.1 默认 absolute

元素布局默认模式改为 absolute，减少重复书写 mode 字段。

### 4.2 flow 自然语言布局（第一期）

第一期只支持最小可用子集：

1. direction: row | column
2. align: start | center | end
3. justify: start | center | end
4. gap: 百分比字符串

不做完整 flex 语法，不做 wrap，不做复杂分布策略。

### 4.3 flow 与子元素 absolute 冲突规则

当父级为 flow，且子级未解绑父级时：

1. 子级可以声明 absolute。
2. 该子级不参与父级 flow 排布。
3. 该子级 absolute 的参考系是父级中心。

---

## 5. 父绑定模型升级

父绑定定义在场景状态层，而不是仅在元素实例层静态定义。

示意：

1. 场景 A：元素 E 绑定 parent=P1。
2. 场景 B：元素 E 绑定 parent=P2。
3. 场景 C：元素 E 解绑 parent，回到根容器参考系。

这样可表达“同一元素在剧情推进中被不同容器接管”的语义。

---

## 6. 编译与运行时职责边界

### 6.1 预解算目标（第一期）

只预解算布局坐标，不预编译动画 from/to。

预解算输出形态：绝对百分比布局参数（例如 left/top/width/height 与锚点）。

### 6.2 运行期职责

运行期只做：

1. 从编译缓存读取目标布局。
2. 按容器尺寸做百分比到像素映射。
3. 应用 transform 和 opacity 插值。

运行期不做复杂 flow 排版，不做结构求解。

### 6.3 缓存失效触发

出现以下变化时必须触发重编译：

1. 场景状态变更。
2. 元素树结构变更。
3. 父绑定关系变更。
4. 比例策略或布局相关配置变更。

## 7. 入场出场动画与增量状态协同

entrance / exit 仍保持“语义解析在 Presentation、执行在 Renderer”的分层。

v2 要求：

1. entrance/exit 解析输入必须基于已增量补全后的目标态。
2. 两帧渲染机制保持不变，继续保证入场动画首帧稳定。

### 7.1 切场景端到端流程索引（实现视角）

本节用于把“切到一个场景时，运行时究竟按什么顺序做事”写清楚，便于评审与后续维护。

说明：机制原理详见：
- `docs/两帧渲染机制.md`

#### 主链路（从 Presentation.setScene 开始）

1) **准备容器尺寸（contentRect）**
- 做什么：同步 content 的 width/height/left/top，并在必要时强制一次 layout flush。
- 意义：避免首次显示/首次播放时 content 尺寸短暂为 0，导致百分比坐标被换算成 (0,0)。

2) **确保 resolved cache 可用**
- 做什么：在 Presentation 侧预计算每个 Scene 的“最终态”（包含跨场景继承 + tombstone）。
- 意义：切场景时只消费缓存结果，避免运行期反复做状态继承推导。

3) **解析并写入本次切场景的 transition 配置**
- 做什么：解析 duration/easing（Scene 优先，未配置回落 Presentation 默认），写入 content.style 与 CSS 变量。
- 意义：保证本次切场景的动画参数在整个渲染链路中一致且可复用。

4) **生成切场景执行计划（SceneChangePlan）**
- 做什么：根据 from/to 场景与方向，解析每个元素的 fromState/toState，并计算最终 delay（含 backward 镜像）。
- 意义：把“动画语义/计划生成”集中在 Presentation（或其纯函数模块）侧；Renderer 只负责执行。

5) **Renderer 执行计划（DOM 与两帧机制）**
- 做什么：
	- 为需要渲染的元素创建/复用 DOM 节点
	- 写入 delay
	- 对入场元素触发两帧机制：Phase 1 写 fromState（禁用过渡）→ reflow → Phase 2 在 rAF 写 toState（恢复过渡）
- 意义：保证入场动画稳定触发，避免浏览器把起始态与目标态合并在同一帧而吞掉动画。

## 8. 导入导出路线

第一期仅实现源码 JSON 导入导出。

### 8.1 目标能力

1. 导出运行时工程为可编辑 JSON。
2. 从 JSON 恢复 Presentation + Element + Scene。
3. 导入后统一走 v2 编译入口。

### 8.2 版本约束

schema 必须包含 version 字段，为后续编译态 JSON 保留演进空间。

---

## 9. 迁移策略（Breaking）

v2 是破坏性升级，重点迁移如下：

1. setState 默认从全量写法变为增量写法。
2. 布局默认模式改为 absolute。
3. 建议旧 demo 迁移到“只写变化量”风格。

迁移文档需提供：

1. 旧写法到新写法对照。
2. replaceState 的使用场景。
3. flow 布局最小示例。
4. 场景级 parent 绑定示例。

---

## 10. 分阶段实施

### Phase 0（已开始）

1. 完成本思维文档。
2. 锁定关键规则与冲突优先级。

### Phase 1

1. 改造 Scene 状态接口（默认增量 + replaceState）。
2. 调整 resolved cache 继承链路。
3. 对齐 entrance/exit 解析输入。

### Phase 2

1. 增加 flow DSL。
2. 建立布局预解算缓存。
3. 运行期改为只消费编译结果。

### Phase 3

1. 源码 JSON 导入导出。
2. schema version 机制。

### Phase 4

1. 重写 demo。
2. 更新 README 与 docs。
3. 补测试与性能基线。

---

## 11. 验收标准

1. 开发者可以在连续场景中只写变化字段，动画与状态正确。
2. row/column + gap 布局可稳定解算为 absolute 百分比坐标。
3. 场景级 parent 绑定切换可正确渲染。
4. resize 后响应式像素映射正确。
5. 导出 JSON 再导入后渲染结果一致。
6. 切场景主线程压力低于旧实现或不高于旧实现。

---

## 12. 非目标

第一期明确不做：

1. 完整 flex 子集。
2. 时间轴关键帧系统。
3. 编译态 JSON 对外协议。
4. 向后兼容层。
